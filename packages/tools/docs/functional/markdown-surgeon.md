# Markdown Surgeon

`md` is the Markdown manipulation CLI and library surface for `dz-tools`.

It edits Markdown by sections today and is the intended owner of the shared Markdown reference resolver described in [`../../../../docs/specs/mrfi.md`](../../../../docs/specs/mrfi.md).

## Existing Section Workflow

The current section commands use exact section IDs generated from headings:

```bash
md outline document.md
md read document.md <section-id>
md write document.md <section-id> "New content"
md append document.md <section-id> "More content"
md remove document.md <section-id>
```

These exact IDs are compact and useful for unchanged headings, but they are not robust enough for durable references after small heading edits.

`md outline --mrfi` adds generated MRFI references while keeping the legacy exact section IDs visible:

```bash
md outline --mrfi document.md
md outline --mrfi --format debug document.md
md outline --mrfi --format base62 document.md
md outline --mrfi --quote document.md
md outline --mrfi --profile full document.md
md ref document.md 12:1-12:32
md ref document.md 12:1-12:32 --format debug
md ref document.md 12:1-12:32 --format base62
md ref document.md 12:1-12:32 --quote
md ref document.md 12:1-12:32 --profile min
```

Generated MRFI references use Hangul compact output by default and the `default` field profile: physical range, structural path, exact fragment hash, fuzzy heading hash, context hashes, and nearby anchor when present. `--profile min` emits only the minimum useful locator fields, and `--profile full` emits every supported safe field, including offset range, fuzzy passage hash, and document hash.

Generated references omit `q=` by default. `--quote` explicitly embeds quote evidence, capped by `--quote-max` with a default of 80 Unicode scalar values. Long quotes are shortened by keeping the beginning, middle, and end. `--format debug` returns the human-readable debug representation, and `--format base62` returns the ASCII compact representation. For `md ref`, source selections use one-based coordinates and an exclusive end column: `startLine:startColumn-endLine:endColumn`.

`md ref` is also the command surface for reference representation conversion:

```bash
md ref '~{v0;r=12:1-12:32;hh=smh64:91a4e8f00c13aa72}' --format debug
md ref '~{v0;r=12:1-12:32;hh=smh64:91a4e8f00c13aa72}' --format base62
md ref '~{v0;r=12:1-12:32;hh=smh64:91a4e8f00c13aa72}' --format hangul
```

Debug, base62, and Hangul conversion are supported for the locator fields generated by `md`: anchors, source ranges, offset ranges, structural paths, exact fragment hashes, fuzzy hashes, context hashes, document hashes, and optional quote evidence.

## Reference Resolution

`md` provides the repository's baseline resolver for Markdown references:

```bash
md resolve document.md ^install_sdk
md resolve document.md ~8F3az91KqP6w
md resolve document.md '~8F3az91KqP6w::previous passage text' ~otherRef
```

Accepted resolver inputs are:

| Form                 | Meaning                                    |
| -------------------- | ------------------------------------------ |
| `^<anchor>`          | Resolve a stable Markdown anchor.          |
| `~<mrfi>`            | Resolve an opaque MRFI locator.            |
| `~<mrfi>::<witness>` | Resolve an MRFI locator with witness text. |

Anchors do not accept witness text. Because `:` is a valid anchor character, `::` inside `^<anchor>` is treated as part of the anchor ID. The `::witness` suffix is a CLI argument convention for MRFI inputs to `md resolve`; it is not part of the MRFI inline syntax, which remains `~<mrfi>` or `~{<mrfi>}`.

`md resolve` accepts multiple inputs in one command and returns one result per input. JSON output includes the resolved passage text when available and must not echo witness text.

Generated MRFI references include `hh=smh64:<hash>` fuzzy heading evidence, `fh=sha256:<prefix>` exact fragment evidence, `ctx=...` context evidence, and `p=...` structural path evidence. The resolver uses `hh` to recover a section when the original range has gone stale and the current document still contains a clearly similar heading scope; it can use `fh` to recover an unchanged passage that moved, `ctx` to recover a changed passage between stable neighbors, and `p` as a structural fallback.

## Text Output

Text output follows the shared reference format:

```text
~8F3az91KqP6w confident 0.86 L42-L47 ^install_sdk
reasons: fuzzy heading match, context suffix match

    ## Installation rapide

    Pour installer le SDK Acme.
```

Passages are indented by four spaces instead of fenced. This avoids ambiguity when the resolved passage itself contains Markdown code fences.

Batch output does not need explicit separators because result headers are non-indented and passages are indented.

## Editing With References

Section editing commands may eventually accept resolved references in addition to exact section IDs, but edit and delete operations must refuse unsafe resolution results.

For write-like commands, the resolver must return `exact` or `confident`. Ambiguous, stale, missing, or invalid references are read-only diagnostics unless a human explicitly chooses a candidate through a future interactive workflow.
